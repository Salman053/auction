WATCHHUB PROJECT DETAILS
========================

1. PROJECT OVERVIEW
WatchHub is a premium Yahoo Japan Auction proxy platform. It allows users to bid on Japanese auctions through a local interface.

2. TECHNOLOGY
- Framework: Laravel 12
- Payments: Stripe & Stripe Connect
- Logic: PHP 8.2, MySQL
- Design: Tailwind CSS 4, Vanilla CSS

3. KEY FEATURES
- Scraping: Real-time search and data sync from Yahoo Japan.
- Bidding: Proxy bidding system with automatic increments.
- Wallet: Multi-currency (Yen) wallet with deposit/withdrawal flows.
- Bidding Power: 500% bidding capacity based on balance.
- Admin Panel: Manage users, audits, scrapers, and payments.

4. SYSTEM FLOWS
- Bidding: User bids -> Capacity checked -> Funds locked -> System updates highest bid.
- Closing: Cron job runs -> Highest bidder wins -> Funds deducted -> Notification sent.
- Withdrawal: User requests -> Admin approves -> Payout processed via Stripe Connect.

5. CURRENT STATUS
The project is in Late Beta. Core logic and UI are done. 
Missing Link: The system currently only tracks internal bids. It needs a browser bot to actually place the bids on the real Yahoo Auctions site.

6. IMPORTANT COMMANDS
- Scrape: php artisan yahoo:scrape "keyword"
- Close Auctions: php artisan auctions:close
- Dev Server: npm run dev / php artisan serve




✦ I have explored the application's core bidding, wallet, and scraping logic. My investigation has revealed several significant bugs and logic gaps that could lead to inconsistent system states and financial
  discrepancies.

  Key Findings & Identified Bugs

   1. Critical: Auction Settlement & Yahoo Price Inconsistency
      The AuctionSettlementService currently assumes the highest internal bidder won the auction without verifying if they were outbid on the actual Yahoo Auctions site. If a Yahoo user bids higher than our internal
  high bidder's maximum, the system will incorrectly mark our user as the winner, potentially leading to incorrect fund deductions and failed orders.

   2. Critical: Missing Scraper Reconciliation
      When the ScrapeYahoo command updates an auction's current_bid_yen from Yahoo, it does not check if this new price outbids our internal users. This leaves the system in a state where a user might still appear as  
  the winning bidder internally even though the Yahoo price has already exceeded their maximum bid.

   3. Standard: Inflated Bidding Capacity with Pending Withdrawals
      The BiddingService calculates a user's 500% bidding capacity incorrectly when they have a pending withdrawal. It subtracts the withdrawal amount 1:1 from the multiplied capacity instead of the base balance. This 
  allows users to place bids far exceeding their intended capacity while a withdrawal is pending.

   4. Standard: Incomplete Proxy Bidding Increments
      The ProxyBiddingService::getIncrement method is missing the higher price tiers used by Yahoo Japan Auctions. For auctions above 50,000 Yen, it uses a flat 1,000 Yen increment, which is incorrect for very
  high-value items (e.g., 100k+, 500k+ Yen tiers).

  Proposed Strategy

  I recommend a phased approach to resolve these issues:

   * Phase 1: Foundation Fixes
       * Fix the bidding capacity calculation in BiddingService.
       * Update ProxyBiddingService with the complete Yahoo increment table.
       * Improve notification logic in BiddingService.

   * Phase 2: Reconciliation Logic
       * Implement a reconciliation process that runs after each scrape. This process will compare Yahoo's current price with internal bids, mark outbid users, and unlock their funds immediately.
       * Update AuctionSettlementService to verify that the internal high bidder's amount actually matches or exceeds the final Yahoo price before declaring a win.

   * Phase 3: Robustness & Verification
       * Add automated tests to reproduce and verify the fixes for the capacity and reconciliation bugs.
       * (Optional but recommended) Increase scraping frequency for auctions ending soon to handle Yahoo's auto-extension feature.

       3. Send an OutbidNotification.
   * Integration: This will be called by the ScrapeYahoo command every time an auction is updated.

  4. Settlement Verification (AuctionSettlementService)

  System Bug Report & Resolution Strategy: WatchHub

  Date: May 11, 2026
  Status: Research Completed / Planning Phase
  Scope: Core Bidding, Wallet Management, and Auction Settlement Logic

  ---

  1. Executive Summary
  An investigation into the WatchHub application has revealed several critical logic errors in the bidding and settlement systems. These bugs primarily stem from a lack of synchronization between internal system states
  and real-time Yahoo Japan Auction data. If left unaddressed, these issues will lead to financial discrepancies, incorrect auction wins, and potential loss of user trust.

  ---

  2. Identified Bugs & Technical Breakdown

  BUG-001: Auction Settlement Inconsistency (Internal vs. External)
   * Description: AuctionSettlementService awards a "Won" status to the highest internal bidder without verifying if they were outbid by a Yahoo user on the actual platform.
   * Impact: CRITICAL. Users will be charged for auctions they did not actually win on Yahoo.
   * Root Cause: settleAuction only looks at internal bids table statuses (active, outbid) and does not check the current_bid_yen synced from Yahoo.

  BUG-002: Missing Scraper Reconciliation
   * Description: The ScrapeYahoo command updates the current_bid_yen from Yahoo but fails to trigger a reconciliation with internal bids.
   * Impact: CRITICAL. Internal high bidders remain marked as "active" (winning) even when the Yahoo price has surpassed their max_amount_yen. Funds remain locked for auctions that are already lost.
   * Root Cause: No post-scrape hook exists to compare the new Yahoo price against the max_amount_yen of internal bidders.

  BUG-003: Bidding Capacity Inflation (Pending Withdrawals)
   * Description: The bidding capacity (500% multiplier) is calculated incorrectly when a user has a pending withdrawal.
   * Impact: STANDARD. Users can place bids using funds that are already designated for withdrawal, leading to potential negative balances if a withdrawal is approved after a bid is placed.
   * Current Logic: (balance * multiplier) - withdrawal_locked
   * Correct Logic: (balance - withdrawal_locked) * multiplier

  BUG-004: Incomplete Increment Table
   * Description: ProxyBiddingService::getIncrement stops at the ¥50,000 tier.
   * Impact: STANDARD. Internal proxy bidding will use incorrect increments for high-value items, leading to price mismatches with Yahoo's actual proxy bidding logic.
   * Root Cause: The method only implements the first four tiers of the Yahoo Japan Auction increment table.

  ---

  3. Proposed Resolution Strategy

  Phase 1: Core Logic Hardening
   1. Correct Bidding Capacity: 
       * Update BiddingService::placeBid to use "Effective Balance" (Balance - Pending Withdrawals) as the base for the 500% multiplier.
   2. Complete Increment Table: 
       * Update ProxyBiddingService::getIncrement to include all Yahoo tiers:
           * ¥50k - ¥100k: ¥1,000
           * ¥100k - ¥500k: ¥3,000
           * ¥500k - ¥1M: ¥5,000
           * Above ¥1M: ¥10,000

  Phase 2: Real-time Reconciliation Engine
   1. Implement AuctionReconciliationService:
       * Create a service that compares the latest Yahoo current_bid_yen with internal max_amount_yen.
       * Logic: If yahoo_price > internal_max_bid, mark internal bid as outbid and immediately unlock wallet funds.
       * Trigger this service from ScrapeYahoo command after each auction update.

  Phase 3: Secure Settlement
   1. Enhance AuctionSettlementService:
       * Add a mandatory verification step in settleAuction.
       * Before awarding a win, check if internal_max_bid >= final_yahoo_price.
       * If not, mark as "Ended - Outbid on Yahoo".

  ---

   * Reproduction Test (Reconciliation): Simulate a scrape update where the Yahoo price exceeds the internal max bid and verify funds are unlocked.
   * Reproduction Test (Settlement): Simulate an auction ending where the internal high bidder is outbid on Yahoo and verify they do not "win" internally.
